#  Security token mechanisms and credential isolation

= Security token mechanisms and credential isolation

In distributed systems, ensuring that sensitive credentials remain isolated while identity and authorization information flow across trust domains is a foundational requirement. Red Hat build of Keycloak leverages robust token mechanisms to provide secure, scalable authentication and authorization.

== Security Token Mechanisms

Red Hat build of Keycloak relies primarily on JSON Web Tokens (JWT) to communicate assertions between parties. These tokens are designed to be compact, URL-safe, and self-contained.

=== Token Types
*   **Access Tokens:** These represent the authorization granted to the client. They contain scopes and user attributes necessary for resource servers to make access control decisions.
*   **Identity Tokens (ID Tokens):** These provide information about the authentication event, such as who the user is and when they authenticated, typically following the OpenID Connect (OIDC) specification.
*   **Refresh Tokens:** Used by clients to obtain new access tokens without requiring the user to re-authenticate, enabling longer-lived sessions without compromising security.

=== Proof of Possession (PoP)
To prevent "bearer token" abuse—where a stolen token can be used by any party—Red Hat build of Keycloak supports **Proof of Possession**. 

When a client uses a token to access a protected resource, it must cryptographically sign the request using its private key. The Resource Server verifies this signature against the fingerprint embedded in the access token. This ensures that even if a token is intercepted, it is useless to an attacker who does not possess the corresponding private key.

== Credential Isolation and Identity Chaining

When applications span multiple trust domains, each with its own authorization server, we must avoid sharing raw user credentials (like passwords) across boundaries. Instead, we use **Identity and Authorization Chaining**.

This approach follows these principles:
1.  **Token Exchange (RFC8693):** Allows a client to exchange a token from one domain for a new token suitable for a different domain.
2.  **JWT Profile (RFC7523):** Provides a mechanism for client authentication using JWTs instead of shared secrets.

=== The Chaining Workflow
In a scenario where a client in *Domain A* needs to access a resource in *Domain B*:

[loweralpha]
. The client in *Domain A* exchanges its existing token with its local authorization server for a JWT authorization grant.
. The local authorization server validates the request and issues the JWT grant.
. The client presents this JWT grant to the authorization server in *Domain B*.
. The authorization server in *Domain B* validates the grant and issues a specific access token for *Domain B*.
. The client uses this new access token to access the protected resource in *Domain B*.

This process isolates the user's primary credentials within *Domain A*. *Domain B* only ever sees an authorization grant that confirms the identity or authority of the client, never the user's original authentication credentials.

== Hands-on Activity: Configuring Token Lifespans

To maintain a strong security posture, you must minimize the time-to-live (TTL) for tokens, ensuring that if a token is compromised, its window of utility is limited.

. **Step 1:** Log in to the Red Hat build of Keycloak Admin Console.
. **Step 2:** Select your Realm from the dropdown menu.
. **Step 3:** Navigate to **Realm settings** > **Tokens**.
. **Step 4:** Configure the following:
*   **Access Token Lifespan:** Set to `5 minutes` (default is usually longer; shortening this increases security).
*   **SSO Session Idle:** Set to `30 minutes`.
. **Step 5:** Click **Save**.

[TIP]
====
Shortening token lifespans forces more frequent validation against the authorization server, but it must be balanced against the performance impact on your network and the overhead on the Keycloak server.
====
