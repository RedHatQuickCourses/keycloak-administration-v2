Role-based and permission-based authorization

Role-based and Permission-based Authorization

In the Red Hat build of Keycloak (RHBK), authorization is the critical process of determining whether a user, application, or service has permission to access specific resources. RHBK manages this by injecting access claims into OIDC tokens or SAML assertions, which applications then parse to make real-time access decisions.

Understanding Role-Based Access Control (RBAC)

RBAC is the primary strategy for managing access in RHBK. It centers on the concept of roles, which act as labels representing a category of users or applications authorized to perform specific tasks.

  • Roles as Categories: Roles group users with shared requirements (e.g., admin, manager, user).

  • Decoupled Permissions: Applications define the permissions associated with a role. By mapping users to these roles, administrators can grant consistent access levels across the organization without configuring individual permissions for every user.

  • Namespaces: RHBK supports two types of role namespaces:

  • Realm Roles: Global roles that apply to users across the entire realm.

  • Client Roles: Roles scoped to a specific application (client), allowing for fine-grained control unique to that service.

User Role Mapping

A user role mapping creates the logical link between a user identity and a specific role.

  • Encapsulation: When a user authenticates, RHBK includes their assigned roles within their security token (JWT/SAML).

  • Efficiency: Because the authorization data travels with the token, the target application does not need to query the identity provider repeatedly, reducing latency and decoupling the application from the identity store.

Leveraging Groups for Scalability

While roles define what a user can do, Groups provide an organizational structure to manage who the users are. * Groups serve as collections of users. * You can assign roles and attributes to a group; any user added to that group automatically inherits those roles. This is a best practice for managing large-scale user populations where individual role assignment becomes administratively burdensome.

Guided Exercise: Authorizing Users with RBAC

This exercise demonstrates how to create a role and map it to a user.

Step 1: Create a Realm Role

  1. Log in to the Admin Console.

  2. Select your realm from the top-left dropdown.

  3. In the left-hand menu, click Realm roles.

  4. Click Create role.

  5. Enter manager as the Role name and click Save.

Step 2: Map the Role to a User

  1. Click Users in the left-hand menu.

  2. Select an existing user (or create a new one).

  3. Select the Role mapping tab.

  4. Click Assign role.

  5. Filter by "Realm roles," select the manager role you created, and click Assign.

Step 3: Verification

  • Observe the user’s details. The user now possesses the manager role, which will be included in their next token issuance, allowing your applications to recognize them as an authorized manager.

Advanced Authorization Strategies

While RBAC is the standard, RHBK supports more sophisticated methods for complex environments:

  • Attribute-Based Access Control (ABAC): Authorization decisions are made based on user or resource attributes (e.g., "Allow access if user.department equals `finance`").

  • Fine-Grained Authorization: RHBK allows you to define policies and permissions directly on resources, scopes, and objects, enabling you to secure specific APIs or UI elements based on runtime conditions.

When planning your authorization architecture, prioritize RBAC for general user-to-application access, and reserve Fine-Grained/Attribute-based controls for highly specific regulatory or data-sensitivity requirements.
