Lab: Authentication and Authorization

Lab: Authentication and Authorization

In this lab, you will implement the core security mechanisms of Red Hat build of Keycloak. You will transition from basic user authentication to granular access control by defining roles and applying them to protected resources.

Objectives

  • Configure a custom Authentication Flow.

  • Define Roles and assign them to users.

  • Implement Role-Based Access Control (RBAC) to authorize user access.

Prerequisites

  • A running instance of Red Hat build of Keycloak.

  • An existing Realm created for your application.

Step 1: Configure Authentication Flows

Keycloak uses "Flows" to determine the sequence of steps a user must complete to be authenticated. In this step, you will modify the flow to enforce specific requirements.

  1. Log in to the Admin Console.

  2. Navigate to Authentication in the left-hand menu.

  3. Click Flows and locate the Browser flow.

  4. Click the Actions menu (three dots) next to the Browser flow and select Duplicate. Name it Custom-Browser-Flow.

  5. Within Custom-Browser-Flow, change the Username/Password Form to Required and add an OTP Form as Required to enforce Multi-Factor Authentication (MFA).

  6. Navigate to Realm Settings > Authentication tab and set the Browser Flow dropdown to Custom-Browser-Flow.

Step 2: Define Roles and Assign Users

Access control in Keycloak is primarily managed through roles.

  1. Navigate to Realm Roles and click Create role.

  2. Create two roles: manager and employee.

  3. Navigate to Users and click Add user. Create a user (e.g., jdoe).

  4. Under the Role mapping tab for jdoe, click Assign role and select employee.

Step 3: Implement Authorization

Now that roles are defined, you will restrict access to a sample application or protected endpoint.

  1. Navigate to Clients and select your target application.

  2. Enable Authorization in the client settings.

  3. Navigate to the Authorization tab for the client:

    • Resource: Define a resource representing your API endpoint (e.g., /api/reports).

    • Scope: Define the allowed actions (e.g., view, edit).

    • Policy: Create a Role-Based Policy. Select the manager role and associate it with the edit scope.

    • Permission: Create a Scope-based permission that links your manager role to the edit scope of the /api/reports resource.

Step 4: Verification

To verify your configuration:

  1. Attempt to log in as jdoe through the application.

  2. Note that the system prompts for the OTP configured in Step 1.

  3. Access the /api/reports endpoint with the edit scope.

  4. Expected Result: The request should be denied (403 Forbidden) because jdoe only possesses the employee role, not the manager role required for editing.

Troubleshooting

  • Token Mismatch: If authorization fails unexpectedly, ensure the Audience mapper is enabled in the Client Scope so the token includes the correct client ID.

  • Flow Not Triggering: Ensure that the Realm Settings specifically point to the duplicated Custom-Browser-Flow rather than the default Browser flow.

  • Logs: Check the Keycloak server logs for WARN messages related to AUTH_FLOW or PERMISSION_DENIED to identify which specific policy rejected the request.
